KG Construction & Code Summarization Pipeline

From Git Repository → Knowledge Graph → KG-aware LLM Summaries

Legend 📁 Repo / Files ⚙️ Script / Step 📄 JSON / CSV Output 🕸 Knowledge Graph 🧠 LLM / Summaries
Stage 0
📁 Repository Setup & Indexing
Motive
Establish a clean, indexed representation of the codebase and verify git clone integrity.
Input
📁 Git repository (source code + full history).
Key scripts
⚙️ step1_verify_clone.py ⚙️ step2_index_repo_files.py
Outputs
  • 📄 file_index.json — all repo files (.py, docs, configs, etc.).
⬇ feeds into
Stage 1
📦 Artifact & Dependency Extraction
Motive
Identify external packages, documents, and archives that influence behavior and provide context.
Input
📄 file_index.json, repository files (code, PDFs, JAR/.whl/.tar.gz, etc.).
Key scripts
⚙️ stepA_generate_manifest.py ⚙️ stepB_tika_extract.py ⚙️ stepC_extract_archives.py ⚙️ stepD_merge_artifacts_into_manifest.py
Outputs
  • 📄 manifest.json — unified inventory of repo artifacts.
  • 📄 artifacts_index.json — extracted packages and versions.
⬇ feeds into
Stage 2
🧩 Code Entity Extraction & Symbol Resolution
Motive
Build a canonical registry of all code entities (functions, classes, methods, variables) with exact locations.
Input
📁 Source code (.py files), 📄 manifest.json, 📄 file_index.json.
Key scripts
⚙️ step3_extract_code_entities.py ⚙️ step3_enrich_code_entities_with_source.py ⚙️ step3_write_full_code_entities.py ⚙️ stepG_build_symbol_table.py
Outputs
  • 📄 code_entities.json, code_entities_full.json — extracted/enriched entities.
  • 📄 symbol_table.json — canonical mapping (qualified_name → definition).
  • 📄 module_index.json, module_defs.json — module-level definitions.
⬇ feeds into
Stage 3
🔗 Call & Dependency Graph Construction
Motive
Model how the codebase is wired: which functions call each other, and how files/modules depend on each other.
Input
📄 symbol_table.json, code_entities_full.json, source code, 📄 manifest.json.
Key scripts
⚙️ stepH_build_call_graph.py ⚙️ stepH_full_call_graph.py ⚙️ stepN_build_file_dependency_graph.py ⚙️ stepM_enrich_call_graph_with_jedi.py ⚙️ stepJ_parse_dependencies.py
Outputs
  • 📄 call_graph.json, call_graph_enriched.json — static call relations.
  • 📄 file_dependency_graph.json, file_dependency_matrix.json.
  • 📄 dependencies.json, dependencies_by_file.json, import_aliases.json.
⬇ feeds into
Stage 4
🧾 Provenance & Metadata Enrichment
Motive
Attach history to code entities: who changed what, when, and through which PRs.
Input
📄 symbol_table.json, code_entities_full.json, git commit & PR history.
Key scripts
⚙️ stepI_map_functions_to_commits.py ⚙️ stepP_enrich_with_commits.py
Outputs
  • 📄 function_commits.json, function_commits_repaired.json — per-function commit info.
  • 📄 commits.json, pull_requests.json, commit_to_prs.json.
  • 📄 nodes_with_commit.json — KG nodes enriched with provenance.
⬇ feeds into
Stage 5
🛠 Configuration & CI/CD Analysis
Motive
Capture deployment context, environments, and possible secrets from configs and CI workflows.
Input
📄 file_index.json, configs (.yaml, .json, .env, Dockerfile, CI workflows, etc.).
Key scripts
⚙️ stepK_parse_ci_workflows.py ⚙️ stepL_parse_configs_and_detect_secrets.py
Outputs
  • 📄 ci_workflows.json, configs.json.
  • 📄 ci_env_secrets_report.json, configs_secrets_report.json.
⬇ feeds into
Stage 6
📚 Dependency Parsing & Resolution
Motive
Extract package-level dependencies from declaration files across languages.
Input
📄 manifest.json, requirements.txt, setup.py, pom.xml, Gemfile, package.json, etc.
Key scripts
⚙️ stepJ_parse_dependencies.py
Outputs
  • 📄 dependencies.json (package-level dependencies).
  • 📄 import_aliases.json reused from Stage 3 for linking imports.
⬇ feeds into
Stage 7
🕸 KG Merge & Schema Construction
Motive
Merge all extracted views (entities, graphs, configs, provenance, deps) into one coherent knowledge graph.
Input
All outputs from Stages 2–6: entities, symbol table, call/dependency graphs, configs, commits & PRs, dependency info.
Key scripts
⚙️ stepO_merge_into_kg.py
Outputs
  • 📄 node_v2.json — canonical KG nodes (id, type, attributes, provenance).
  • 📄 edges.json — KG edges (source, target, relation type, reasons).
  • 📄 kg_schema.json — KG ontology (node types, edge types, properties).
⬇ feeds into
Stage 8
🧠 Node Enrichment & Disambiguation
Motive
Improve node quality: resolve ambiguous references, backfill missing info, and attach richer semantics.
Input
📄 node_v2.json, edges.json, AST data, call graph data, text search results.
Key scripts
⚙️ stepQ_enrich_nodes_extended.py ⚙️ helper disambiguation / backfill scripts
Outputs
  • 📄 nodes_resolved.json — nodes with precise file/line locations.
  • 📄 nodes_disambiguated.json — ambiguous references resolved.
  • 📄 nodes_ast_backfilled.json, nodes_callgraph_backfilled.json, nodes_textsearch_backfilled.json.
⬇ feeds into
Stage 9
📤 Final KG Export & QA
Motive
Convert the KG into consumable formats (CSV/JSON/graph DB) and produce a QA report.
Input
📄 node_v2.json, edges.json, kg_schema.json, enriched node variants.
Outputs
  • 📄 kg_nodes.csv, kg_edges.csv — tabular exports of KG.
  • 📄 kg_report.json — node/edge counts, coverage, warnings.
  • 🕸 FINAL KG — loaded into Neo4j / RDF store / JSON index.
⬇ used by
Stage 10
🧠 → 📝 KG-Aware Code Summarization
Motive
Use KG facts as grounded context so the LLM produces accurate, explainable code summaries.
Input
🕸 FINAL KG (nodes & edges), 📁 source code snippet, user query or function of interest.
Key artifact & outputs
  • 📄 summarizer_request.json — packaged KG context (callers, callees, commits, configs, deps).
  • 🧠 LLM uses this structured context →
  • 📝 Generated code summaries that are grounded, traceable, and explainable.

This view is meant for docs: each stage is independently understandable, but the arrows show the end-to-end flow from raw repo → enriched KG → LLM-based summarization.
